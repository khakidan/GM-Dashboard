### 🔴 Bugs to Fix

**`DeathOverlay.tsx` — missing `currentTime = 0` reset causes a stuck video frame on a second death event while the overlay is still showing.** Confirmed via direct trace, not assumed: `useOverlayEvent`'s `fire()` overwrites `deathEvent` directly from one non-null value to another (never passes through `null`/`clearValue` in between), so if a GM triggers a death for Character B while Character A's `DeathOverlay` is still on screen, React reuses the same component instance (same type, no `key` prop) rather than unmounting/remounting. `DeathOverlay`'s video-play `useEffect` has an empty dependency array and no `currentTime = 0` reset (unlike the other 5 overlays, all of which have both), so it never re-runs on the second event — the character name text updates, but the blood-impact video stays frozen on the first death's final frame instead of replaying. Found while investigating overlay componentization; scoped to be fixed as part of that same work, since the fix (adding a real dependency array and the reset) naturally falls out of building the shared video-layer hook all 6 overlays will use.

**The full-app-state-rollback pattern (already fixed 3 times this session in `useEncounters.ts`/`useParty.ts`/`useNpcLibrary.ts`) is confirmed present in 11 more handler functions across 4 additional files, plus a worse variant (rollback entirely missing) in a 5th.** Found while auditing for repeated hook patterns, then explicitly verified file-by-file (not assumed from surface similarity) — every one of the following captures the *entire* app state via `getSnapshot()` before an optimistic update, then on failure calls `updateState(() => <that whole snapshot>)`, discarding any unrelated concurrent state changes elsewhere in the app during the async write window — the identical bug already fixed elsewhere this session, confirmed to still be present here:
- **`useBatchActions.ts`** — 5 handlers: `updateCombatant` (a separate, duplicate local implementation of this name — not the same function as `useCombatantMutations.ts`'s), `handleApplyMultiDamage`, `handleApplyMultiHealing`, `handleApplyMultiCondition`, `handleDeleteSelected`.
- **`useDeathSaves.ts`** — 2 functions: `recordDeathSave`, `applyDamageToUnconscious`.
- **`useEncounterPresetLoader.ts`** — 2 functions: `handleAddPreset`, `handleAddNpc`.
- **`useCombatantMutations.ts`** — 2 functions: `removeCombatant`, `updateCombatant`.

**Investigation confirmed something bigger than a rollback-scope issue: `useBatchActions.ts` and `useCombatantMutations.ts` each independently reimplement the entire `updateCombatant` logic wholesale** — not just a shared rollback pattern, but two genuinely separate ~100–150 line implementations of the same optimistic-update-plus-DB-write-dispatch-plus-rollback logic, confirmed via direct side-by-side reading of both (neither delegates to or wraps the other). This is a bigger duplication finding than anything the earlier utility-function audit surfaced. `useHealthChange.ts`'s `handleHealthChange` receives `updateCombatant` as a dependency-injected function argument (typed to return `void`), fires it without awaiting, and — critically — it's `useBatchActions.ts`'s own local `updateCombatant` that actually gets called this way in `handleApplyMultiDamage`/`handleApplyMultiHealing`'s loops, not `useCombatantMutations.ts`'s.

**Traced the full failure chain concretely**: when a batch of 5 combatants' DB writes fail, `useBatchActions.ts`'s local `updateCombatant` catch block fires independently for each of the 5 concurrent failures, each one calling `updateState(() => previousState)` with its own captured snapshot — 5 tasks racing to overwrite the entire global store, causing genuine state tearing, while `handleApplyMultiDamage`'s own outer catch block never fires at all (the loop already completed synchronously and successfully before any of the 5 async failures could occur).

**Decision made: consolidate to one shared `updateCombatant` implementation** rather than patch both rollbacks independently — this resolves the duplication and the racing-rollback hazard in one move, since there'd only be one rollback mechanism left to reason about instead of two independently racing ones. **Confirmed via full side-by-side comparison of both real implementations: `useCombatantMutations.ts`'s version is a genuine strict superset of `useBatchActions.ts`'s local copy, safe to consolidate onto directly.** The local `useBatchActions.ts` version is missing real, user-facing behavior the canonical version already has: condition-based AC recalculation, exhaustion HP-max capping, condition-timer cleanup, initiative re-sorting, combat-log events for condition changes, legendary-action/recharge persistence, and Rage overlay firing. This means `handleApplyMultiCondition` (which routes through the incomplete local version) currently has a **real, silent rules-accuracy gap**: bulk-applying "Raging" to multiple PCs at once doesn't fire the Rage overlay or recalculate AC the way applying it to one combatant at a time already correctly does — consolidating fixes this as a direct side effect, not just removing duplication.

Confirmed all real call sites for both: the local `useBatchActions.ts` version has exactly 3 (via `handleHealthChange` in the Damage/Healing loops, and directly in `handleApplyMultiCondition`); the canonical `useCombatantMutations.ts` version has 5 (`ActiveEncounterTab/index.tsx`'s `onUpdateCombatant` prop and its `useHealthChange` wiring, `useCombatTurn.ts`, `useCombatConcentration.ts`, `useEncounterPresetLoader.ts`).

### 🔵 Architecture / Technical Debt

**Remaining Technical Debt:** None.

**Future audit categories, discussed but not yet started** — beyond bug-hunting and componentization (Phase 1/2 above), a professional React/TypeScript codebase review typically also covers several categories this project hasn't yet run as a dedicated pass:

- **Type safety** — a systematic sweep for `any`/unsafe type casts, missing null checks, and places where TypeScript's strictness is being worked around rather than honored. Only ever caught incidentally so far (e.g. the `as any` shortcut proposed and rejected during the `useSettings.ts` type-safety fix, then properly resolved via a narrow, justified cast instead).
- **Accessibility (a11y)** — semantic HTML, ARIA attributes, keyboard navigation, focus management. Also only ever caught incidentally (`IconButton`/`CardHeaderChevron` needing real `aria-label`s, `Tabs.tsx` needing genuine `role="tablist"` + arrow-key navigation, `Accordion.tsx` needing to be a real `<button>` instead of a `div` with manually-reimplemented keyboard handling) — a dedicated pass would likely find more.
- **Security** — this app handles real OAuth tokens and writes to a GM's actual Google Sheets, so this matters more than average for an app this size. Worth checking `dangerouslySetInnerHTML` usage, how tokens are stored/read, and injection risk from user-generated content (NPC names, notes, etc.) rendered without sanitization. One real OAuth CSRF bug was already found and fixed this session (see `CHANGELOG.md`) — a dedicated security pass might find more in the same vein.
- **Performance** — unnecessary re-renders and missing memoization, checked carefully rather than blanket-applying `useMemo`/`useCallback` everywhere (this session already learned that lesson directly — the `useAppState.ts`/`useMoodPresets.ts` fixes were about correcting genuinely broken memoization, not adding memoization indiscriminately). Also worth checking for expensive computations re-running on every render unnecessarily.
- **Error handling consistency** — server-side error-response shaping was already audited (Category 6 above), but client-side async error handling hasn't been checked for consistency — are failures always caught, and always surfaced to the GM the same way, or does the pattern vary file to file?
- **Code smells beyond duplication** — magic numbers/strings that should be named constants, deeply nested conditionals, dead code or unused exports, "god functions" doing too many unrelated things (a function can be short in line count and still be doing 5 unrelated things at once — a different problem than file size, which Phase 2's census already covers).

Not yet scoped as concrete audit prompts — to be run the same way as the componentization audit (isolated, evidence-required categories) once prioritized.
