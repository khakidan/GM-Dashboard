**`components/` bug-hunt audit findings — `ActiveEncounterTab/` complete, verified with spot-checks (not exhaustive).** All ~30 files in this directory (including `hooks/`) were reported "CLEAN" by AI Studio in a single pass. Given the size, not every quote was independently verified — but spot-checking surfaced one real, meaningfully mislabeled bug, which argues for treating the rest of this "all clean" batch with some caution rather than full trust:

1. **`useCombatLifecycle.ts` — NPC initiative incorrectly floored at 1, mislabeled as "RAW-correct."** `const total = Math.max(1, d20Roll + dexMod);` clamps initiative to a minimum of 1 — but 5e RAW has no such floor; a creature with a strongly negative DEX modifier can legitimately roll 0 or negative initiative. AI Studio's audit quoted this exact line as evidence of a correct implementation ("Initiative roll bound limits RAW"), which is backwards — the clamp itself is the bug. Compounding issue: the toast notification displays the clamped `total` alongside the *unclamped* `d20Roll`/`dexMod` breakdown text, so the displayed arithmetic won't add up whenever the clamp actually changes the result (e.g. "Rolled initiative for Goblin: 1 (d20: 3 + DEX −5)" where 3 + (−5) = −2, not 1).

Spot-checked and confirmed accurate (not a bug): `useCombatantExpanded.ts`'s `handleExhaustionDeath` — correctly marks a PC deceased at exhaustion level 6, matching real 5e RAW, quote matched the real file exactly.

Additionally verified 3 higher-risk files not part of AI Studio's original quoted findings, to sanity-check the "all clean" batch further:

2. **`useBatchActions.ts` — batch rollback logic is largely unreachable for the failure case it appears to guard against.** `handleHealthChange` (from `useHealthChange.ts`) is not async and never awaits its own internal `updateCombatant` calls — fire-and-forget. Each batch loop (`handleApplyMultiDamage`/`handleApplyMultiHealing`/`handleApplyMultiCondition`) calls `handleHealthChange` per selected combatant without awaiting it, then finishes the loop and fires `toast.success(...)` before the underlying async DB writes have necessarily resolved. If one combatant's write fails later, it's caught only by `updateCombatant`'s own internal try/catch (which reverts just that one combatant and shows its own separate error toast) — potentially well after the batch already reported success. The batch-level `catch (err) { updateState(() => previousState); toast.error(...) }` in each handler can only really catch synchronous errors thrown directly in the loop body, not the async DB failures it looks like it's designed to protect against.

`useCombatSync.ts` and `index.tsx` (the tab's main orchestrating component) were also checked directly and are genuinely clean — no dependency-array issues, correct `key={c.id}` (not index) on the reorderable combatant list, correct event listener cleanup.

3. **Round-counter desync between `combatState.round` and `activeCombatLog.currentRound` — real, confirmed, decision made, ready to prompt.** Investigated in depth per Dan's request since round accuracy matters. `initCombatLog` (`dashboardStore.ts`) always hardcodes `currentRound: 1`, but it's only called from `handleCallInitiative` (`useCombatLifecycle.ts`) — a separate, manually-triggered action, not something automatically tied to combat starting. Meanwhile `combatState.round` is set to `1` immediately in `startEncounter` (`useEncounterLifecycle.ts`, which — separate documentation bug — `file-reference.md` incorrectly credits with calling `initCombatLog`/`advanceCombatLogRound`/`clearCombatLog`; it calls none of them) and advances on every `nextTurn()` regardless of whether `activeCombatLog` exists yet. If "Call for Initiative" is clicked late (after `combatState.round` has already advanced past 1), `initCombatLog` resets the log's round tracking back to 1 while the real round counter is already ahead — genuinely desynced from that point on. Confirmed worse via `useEncounterResume.ts`: resuming an in-progress encounter restores `combatState.round` from the sheet (could be any value) but never touches `activeCombatLog` at all, and `activeCombatLog` is only persisted to `localStorage`, never to the sheet (no such column in `schema.md`) — so **any cross-device resume, or resume after cleared browser storage, of a multi-round encounter hits this desync as a matter of course**, not just as a narrow edge case. **Decision made (Option A): stop treating `activeCombatLog.currentRound` as an independent counter — sync it to `combatState.round` instead of a schema change (Option B, persisting `activeCombatLog` to the sheet, was considered and rejected as heavier than needed).** Fix plan when we pick this up: change `initCombatLog`'s signature to take a `startingRound` parameter instead of hardcoding `1`, and update its one call site in `handleCallInitiative` to pass the current `combatState.round`. Search for other call sites of `initCombatLog` first in case there are more than the one known site. Not yet prompted — deliberately deferred to keep moving through directory audits first.

**`hooks/` re-verification pass — the 11 remaining previously-unverified "clean" files, now actually checked. 4 real findings, most significant one app-wide.**

1. **`useAppState.ts` — the primary selector rebuilds a brand-new object on every call, with no equality function, causing app-wide unnecessary re-renders on every single store update.** `useDashboardStore((s): AppState => ({ characters: s.characters, ... }))` reconstructs a fresh object literal every time, and without `shallow`/`useShallow`, Zustand's default `Object.is` check always sees a new reference — so every component using `useAppState()` (used pervasively across the whole app) re-renders on every store update, even ones to fields this selector doesn't even include (like `activeCombatLog`, which lives in the same store but isn't part of `AppState`).
