### 🔵 Architecture / Technical Debt

**Remaining Technical Debt:**

- **`abilityScores.ts` and `spellcasting.ts` have a circular import.** `abilityScores.ts`'s `parseProficiencies()` calls `parseSpellcastingAbility()` from `spellcasting.ts`; `spellcasting.ts` needs `abilitiesInOrder`/`AbilityName` from `abilityScores.ts`. Confirmed currently safe — every cross-import is used only inside function bodies, never at either file's top level, so ES modules' live-binding semantics handle this correctly with no runtime issue. Not pre-existing to any recent change; surfaced while relocating `abilitiesInOrder` to its correct domain home. A genuine fix would require either moving `parseSpellcastingAbility`/`SpellcastingAbility` into `abilityScores.ts` (a domain mismatch — that logic is conceptually about spellcasting, not ability scores) or restructuring `parseProficiencies()` to no longer call into `spellcasting.ts` directly (would change a working function's signature and require updating every caller). Deliberately deferred — real complexity/risk for a purely architectural benefit, not a functional one.
- **54 pre-existing TypeScript errors across 18 test files, surfaced by a real `npx tsc --noEmit` run (the default, broader config) while verifying an unrelated fix.** Not caused by, or related to, any change made this session — the same run against `npx tsc -p tsconfig.build.json --noEmit` (the actual build-scoped config) was completely clean. Every one of the 54 is a test-file type mismatch: mock/fixture objects missing properties that real interfaces now require (`Encounter` missing `status`/`difficultyName`, `Combatant` missing `ac`/`maxHp`/`currentHp`/`passivePerception`, `CombatState` missing `combatStarted`/`actionContext`, `NPC` missing/gaining fields, a `deleteDimension` property referenced that doesn't exist on `BatchRequest`, `WriteValue` not assignable to `string` in a few sheet-write assertions, 2 unused `@ts-expect-error` directives). Likely accumulated gradually as real interfaces evolved without every test fixture being updated to match — worth a real triage pass at some point, but out of scope for anything done this session.

**Future audit categories, discussed but not yet started** — beyond bug-hunting and componentization (Phase 1/2 above), a professional React/TypeScript codebase review typically also covers several categories this project hasn't yet run as a dedicated pass:

- **Type safety** — a systematic sweep for `any`/unsafe type casts, missing null checks, and places where TypeScript's strictness is being worked around rather than honored. Only ever caught incidentally so far (e.g. the `as any` shortcut proposed and rejected during the `useSettings.ts` type-safety fix, then properly resolved via a narrow, justified cast instead).
- **Accessibility (a11y)** — semantic HTML, ARIA attributes, keyboard navigation, focus management. Also only ever caught incidentally (`IconButton`/`CardHeaderChevron` needing real `aria-label`s, `Tabs.tsx` needing genuine `role="tablist"` + arrow-key navigation, `Accordion.tsx` needing to be a real `<button>` instead of a `div` with manually-reimplemented keyboard handling) — a dedicated pass would likely find more.
- **Security** — this app handles real OAuth tokens and writes to a GM's actual Google Sheets, so this matters more than average for an app this size. Worth checking `dangerouslySetInnerHTML` usage, how tokens are stored/read, and injection risk from user-generated content (NPC names, notes, etc.) rendered without sanitization. One real OAuth CSRF bug was already found and fixed earlier this session — a dedicated security pass might find more in the same vein.
- **Performance** — unnecessary re-renders and missing memoization, checked carefully rather than blanket-applying `useMemo`/`useCallback` everywhere (this session already learned that lesson directly — the `useAppState.ts`/`useMoodPresets.ts` fixes were about correcting genuinely broken memoization, not adding memoization indiscriminately). Also worth checking for expensive computations re-running on every render unnecessarily.
- **Error handling consistency** — server-side error-response shaping was already fully audited and fixed (see `CHANGELOG.md`), but client-side async error handling hasn't been checked for consistency — are failures always caught, and always surfaced to the GM the same way, or does the pattern vary file to file?
- **Code smells beyond duplication** — magic numbers/strings that should be named constants, deeply nested conditionals, dead code or unused exports, "god functions" doing too many unrelated things (a function can be short in line count and still be doing 5 unrelated things at once — a different problem than file size, which Phase 2's census already covers).

Not yet scoped as concrete audit prompts — to be run the same way as the componentization audit (isolated, evidence-required categories) once prioritized.
